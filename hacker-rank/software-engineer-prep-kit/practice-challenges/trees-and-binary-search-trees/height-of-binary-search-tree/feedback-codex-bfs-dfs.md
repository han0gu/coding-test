# DFS와 BFS 관점 정리

## 이 문제에서 헷갈렸던 지점

`Height of Binary Search Tree` 문제는 트리의 높이를 구하는 문제다.

현재 풀이에서는 루트 노드를 꺼낸 뒤 자식 노드를 넣고, 다시 그중 하나를 꺼내 자식 노드를 넣는다. 그래서 겉으로 보면 "노드를 하나씩 꺼내면서 자식을 추가하니까 BFS 아닌가?"라고 느낄 수 있다.

하지만 DFS와 BFS를 나누는 기준은 `자식을 넣는지 여부`가 아니다. 둘 다 현재 노드를 처리한 뒤 다음에 방문할 후보로 자식을 넣는다.

기준은 넣는 순서가 아니라 `꺼내서 실제로 방문하는 순서`다.

여기서 `방문한다`는 것은 `stack`이나 `queue`에서 꺼내 그 노드의 자식을 `확인하고 처리하는 순간`을 뜻한다.

따라서 DFS/BFS를 판단할 때는 이렇게 봐야 한다.

```text
1. 현재 노드를 처리한 뒤 어떤 노드들이 후보로 남는가?
2. 그 후보들 중 다음에 실제로 꺼내 방문되는 노드는 무엇인가?
```

핵심 차이는 **이미 후보로 올라와 있는 노드들 중 무엇을 다음에 꺼내 방문하느냐**다.

## 이 풀이가 DFS인 이유

현재 풀이의 핵심 구조는 다음과 같다.

```python
stack = [(0, 1)]

while stack:
    i, depth = stack.pop()

    if leftChild[i] != -1:
        stack.append((leftChild[i], depth + 1))
    if rightChild[i] != -1:
        stack.append((rightChild[i], depth + 1))
```

이 풀이는 재귀를 쓰지 않았지만 DFS다.

이유는 `stack`에 남겨둔 후보 중에서 최근에 발견한 노드를 먼저 **꺼내 방문하기** 때문이다. 즉, 현재 선택한 가지 아래에서 새로 발견한 자식들을 먼저 따라가고, 이전 갈림길에서 남겨둔 형제 노드는 나중에 처리한다.

DFS의 본질은 다음과 같다.

```text
현재 경로를 가능한 한 깊게 따라간다.
더 내려갈 수 없으면 이전 갈림길로 돌아와 다른 경로를 본다.
```

이 문제의 스택 풀이는 정확히 이 흐름이다. 루트에서 시작해 한 자식을 선택하면, 같은 깊이의 다른 형제 노드를 바로 처리하기보다 선택한 자식의 아래 방향으로 더 내려간다.

## 넣는 순서와 빼는 순서를 분리해서 보기

헷갈릴 때는 `append`하는 순서와 `pop`해서 방문하는 순서를 나눠서 보면 좋다.

- 넣는 순서: 같은 부모의 자식 중 왼쪽을 먼저 볼지, 오른쪽을 먼저 볼지를 바꾼다.
- 빼는 순서: 탐색이 깊이 우선인지, 너비 우선인지를 결정한다.

예를 들어 현재 코드처럼 왼쪽 자식을 먼저 넣고 오른쪽 자식을 나중에 넣으면:

```python
stack.append(left)
stack.append(right)
```

스택 안에서는 오른쪽 자식이 더 나중에 들어갔으므로 오른쪽 자식이 먼저 꺼내져 방문된다.

반대로 오른쪽 자식을 먼저 넣고 왼쪽 자식을 나중에 넣으면:

```python
stack.append(right)
stack.append(left)
```

왼쪽 자식이 먼저 꺼내져 방문된다.

하지만 두 경우 모두 DFS다. 왼쪽부터 방문하느냐 오른쪽부터 방문하느냐만 달라질 뿐, 한 자식을 꺼낸 뒤에는 그 자식의 아래 방향으로 계속 내려가기 때문이다.

즉, `append` 순서는 **DFS 안에서의 좌우 방문 순서**를 바꾼다. DFS인지 BFS인지를 가르는 기준은 `append` 순서가 아니라, 후보 중에서 다음 노드를 어떤 방식으로 꺼내 실제 방문하느냐다.

## "자식을 둘 다 넣는데 왜 DFS인가?"

예제 트리를 보자.

```text
        4
      /   \
     2     6
    / \   / \
   1   3 5   7
```

루트 `4`를 처리하면 자식 `2`, `6`이 후보가 된다.

```text
처리 완료: 4
후보: 2, 6
```

여기까지만 보면 DFS와 BFS가 같아 보인다. 둘 다 `4`의 자식을 후보에 넣기 때문이다.

차이는 그 다음부터다.

만약 다음에 `2`를 처리했다고 해보자. 그러면 `2`의 자식 `1`, `3`이 후보에 추가된다.

```text
처리 완료: 4, 2
후보: 6, 1, 3
```

이때 DFS는 `1` 또는 `3`처럼 방금 내려간 `2`의 자식을 먼저 꺼내 방문한다. 아직 같은 깊이에 있던 `6`은 잠시 미뤄둔다.

반면 BFS라면 `1`, `3`을 보기 전에 같은 깊이의 `6`을 먼저 꺼내 방문해야 한다.

```text
DFS 관점: 4 -> 2 -> 3 -> 1 -> 6 ...
BFS 관점: 4 -> 2 -> 6 -> 1 -> 3 -> 5 -> 7
```

즉, DFS와 BFS의 차이는 "자식을 넣었는가"가 아니라 **후보 중에서 같은 레벨의 형제를 먼저 꺼내 방문하는가, 아니면 방금 선택한 가지의 자식을 먼저 꺼내 더 내려가는가**다.

## 현재 코드의 실제 방문 순서

현재 코드처럼 왼쪽 자식을 먼저 넣고 오른쪽 자식을 나중에 넣으면:

```python
if leftChild[i] != -1:
    stack.append((leftChild[i], depth + 1))
if rightChild[i] != -1:
    stack.append((rightChild[i], depth + 1))
```

`stack.pop()`은 마지막에 넣은 값을 먼저 꺼낸다. 따라서 예제 트리에서는 보통 오른쪽 자식부터 방문한다. 여기서 중요한 판단 기준은 "오른쪽 자식을 나중에 넣었다"가 아니라, 그 결과 **오른쪽 자식이 다음 방문 노드로 먼저 꺼내진다**는 점이다.

방문 순서는 대략 다음과 같다.

```text
4 -> 6 -> 7 -> 5 -> 2 -> 3 -> 1
```

왼쪽부터 방문하고 싶다면 오른쪽 자식을 먼저 넣고 왼쪽 자식을 나중에 넣으면 된다.

하지만 방문 순서가 왼쪽 우선인지 오른쪽 우선인지는 DFS/BFS 구분의 핵심이 아니다. 중요한 것은 `6`을 선택해 방문한 뒤에는 `6`의 아래로 더 내려가고, `2`는 나중에 꺼내 방문한다는 점이다. 이것이 깊이 우선 탐색의 성격이다.

## BFS라면 어떻게 달라지는가?

BFS는 레벨을 보존한다.

```text
깊이 1: 4
깊이 2: 2, 6
깊이 3: 1, 3, 5, 7
```

BFS에서는 깊이 2에 있는 `2`, `6`을 모두 처리한 뒤에야 깊이 3의 `1`, `3`, `5`, `7`로 넘어간다.

그래서 BFS는 보통 큐를 사용한다.

```python
from collections import deque

queue = deque([(0, 1)])

while queue:
    i, depth = queue.popleft()

    if leftChild[i] != -1:
        queue.append((leftChild[i], depth + 1))
    if rightChild[i] != -1:
        queue.append((rightChild[i], depth + 1))
```

여기서는 먼저 후보가 된 노드를 먼저 꺼내 방문한다. 따라서 루트의 자식들이 모두 처리된 뒤에 그 아래 레벨로 내려간다.

## `queue`와 `deque`의 차이

`queue`는 보통 자료구조의 개념을 말한다.

```text
먼저 들어간 값이 먼저 나온다.
First In, First Out
```

반면 `deque`는 Python에서 사용할 수 있는 구체적인 자료구조다. 이름은 `double-ended queue`의 줄임말이고, 양쪽 끝에서 값을 넣고 뺄 수 있다.

```python
from collections import deque

dq = deque()

dq.append(x)      # 오른쪽에 넣기
dq.pop()          # 오른쪽에서 빼기

dq.appendleft(x)  # 왼쪽에 넣기
dq.popleft()      # 왼쪽에서 빼기
```

BFS에서는 앞에서 꺼내고 뒤에 넣는 흐름이 필요하다.

```python
queue = deque([(0, 1)])

while queue:
    i, depth = queue.popleft()
    # 자식은 queue.append(...)로 뒤에 추가
```

이렇게 하면 먼저 후보가 된 노드부터 꺼내 방문하므로 레벨 순서가 유지된다.

## 왜 `list.pop(0)`은 느린가?

Python `list`로도 겉보기에는 큐처럼 사용할 수 있다.

```python
queue = [1, 2, 3, 4]
queue.pop(0)  # 1
```

하지만 `list.pop(0)`은 `O(n)`이다. 맨 앞 원소를 제거한 뒤, 뒤에 있던 원소들을 한 칸씩 앞으로 당겨야 하기 때문이다.

```text
제거 전:
index:  0  1  2  3
value: [1, 2, 3, 4]

pop(0) 후:
index:  0  1  2
value: [2, 3, 4]
```

이 상태를 만들려면 내부적으로 `2`, `3`, `4`의 위치가 앞으로 이동해야 한다.

```text
2 -> index 0
3 -> index 1
4 -> index 2
```

BFS처럼 앞에서 계속 꺼내는 작업을 반복하면 이 비용이 누적된다. 그래서 Python에서 BFS용 큐는 보통 `list`가 아니라 `collections.deque`를 사용한다.

## 시작 위치만 바꾸면 안 되나?

개념적으로는 가능하다. C/C++에서 포인터나 인덱스를 이용해 큐의 시작 위치만 바꾸는 방식이 바로 그런 접근이다.

예를 들어 논리적으로는 이렇게 생각할 수 있다.

```text
실제 저장 공간: [1, 2, 3, 4]
head -> 1

앞에서 하나 제거한 뒤:
실제 저장 공간: [1, 2, 3, 4]
head -> 2
```

이 방식이라면 실제 원소를 모두 앞으로 당기지 않아도 된다. 큐를 직접 배열로 구현할 때는 이런 `head` 인덱스 방식을 자주 쓴다.

하지만 Python의 `list.pop(0)`은 그렇게 동작하지 않는다. Python `list`는 일반적인 동적 배열처럼 빠른 인덱스 접근과 연속된 저장 구조를 우선하는 자료구조다. `list[0]`은 항상 현재 리스트의 첫 번째 원소여야 하므로, 앞 원소를 제거하면 뒤 원소들을 앞으로 당기는 방식으로 리스트를 정리한다.

앞쪽 제거를 빠르게 하도록 만든 자료구조가 `deque`다. 따라서 BFS처럼 앞에서 자주 꺼내야 하는 문제에서는:

```text
list.pop(0)       -> O(n)
deque.popleft()   -> O(1)
```

로 보는 것이 안전하다.

정리하면, "시작 위치만 바꾸는 큐"는 가능한 구현이다. 다만 Python `list`가 그 방식으로 구현된 큐가 아니고, Python에서는 그 용도로 `deque`를 쓰는 것이 표준적인 선택이다.

## 면접에서 설명할 때의 관점

면접에서 단순히 "`stack`을 쓰면 DFS고 `queue`를 쓰면 BFS입니다"라고만 말하면 외운 답처럼 들릴 수 있다.

더 좋은 설명은 탐색의 진행 방식으로 말하는 것이다.

> 이 풀이는 DFS입니다. 판단 기준은 자식을 어떤 순서로 넣었는지가 아니라, 후보 노드들 중 무엇을 다음에 꺼내 실제로 방문하느냐입니다. 이 코드는 명시적인 스택에 아직 방문하지 않은 노드를 저장해두고 `pop()`으로 하나씩 꺼내 방문합니다. 현재 노드의 자식을 후보에 넣은 뒤, 같은 레벨의 다른 노드를 모두 처리하기 전에 방금 선택한 가지 아래로 계속 내려가므로 깊이 우선 탐색입니다. 즉, 재귀 DFS를 Python 재귀 제한 없이 반복문과 스택으로 바꾼 iterative DFS라고 볼 수 있습니다.

BFS와 비교해서는 이렇게 말할 수 있다.

> BFS라면 같은 깊이의 노드들을 모두 꺼내 방문한 뒤 다음 깊이로 넘어갑니다. 예를 들어 루트 `4` 다음에는 깊이 2의 `2`, `6`을 모두 방문하고 나서야 `1`, `3`, `5`, `7`을 봅니다. 하지만 이 풀이는 한 자식을 선택해 방문하면 그 자식의 아래 방향으로 먼저 내려가므로 BFS가 아니라 DFS입니다.

## 이 문제에서는 DFS와 BFS 모두 가능하다

이 문제의 목표는 루트에서 리프까지의 최대 노드 수, 즉 트리 높이를 구하는 것이다.

DFS로 풀면 각 노드와 현재 깊이를 함께 들고 다니면서 방문한 깊이의 최댓값을 갱신하면 된다.

```text
노드를 방문할 때마다 answer = max(answer, depth)
```

BFS로 풀어도 가능하다. 레벨 순서대로 탐색하면서 마지막으로 도달한 깊이를 기록하면 된다.

다만 현재 풀이는 `stack.pop()`으로 한 경로를 먼저 깊게 따라가는 구조이므로 DFS다. 이 문제에서는 DFS/BFS 중 무엇을 쓰느냐보다, 각 노드를 한 번씩만 방문하고 현재 깊이를 정확히 관리하는 것이 더 중요하다.

## 핵심 정리

- 재귀를 쓰면 자동으로 DFS가 되는 경우가 많다. 함수 호출 스택이 경로를 기억하기 때문이다.
- 직접 만든 `stack`을 쓰는 반복문도 DFS가 될 수 있다.
- DFS의 핵심은 한 경로를 깊게 따라간 뒤 되돌아오는 것이다.
- BFS의 핵심은 같은 깊이의 노드를 모두 처리한 뒤 다음 깊이로 넘어가는 것이다.
- DFS/BFS는 자식을 넣는 순서만 보고 판단하지 않는다.
- 후보 중 어떤 노드가 다음에 꺼내져 실제 방문되는지를 봐야 한다.
- 현재 풀이처럼 `stack.pop()`으로 최근에 발견한 자식을 먼저 꺼내 방문하면 DFS다.
- `queue.popleft()`로 먼저 발견한 노드부터 꺼내 방문하면 BFS다.
